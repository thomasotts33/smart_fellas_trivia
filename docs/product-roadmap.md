## Build Philosophy

1. **Each phase ships something usable.** No phase leaves the app broken. After any phase, you can demo what you have built.
2. **Infrastructure before features.** The first phase sets up the app, API, database, auth skeleton, and dev workflow so feature work is clean.
3. **Magic moment first.** The product earns its keep when logged games become visual progress. The roadmap reaches that by the end of Phase 2.
4. **Test as you go.** Scoring rules are product-critical, so scoring and validation get automated tests early.
5. **Progressive enhancement.** Start with manual entry and reliable charts, then add team polish and later scaling.
6. **Review before proceeding.** Push each completed phase as a PR and let a review agent such as CodeRabbit review it before starting the next phase.
7. **Respect game night.** Do not build live scoring before the post-game workflow works beautifully.
8. **Keep team tenancy from day one.** Even the single-team MVP should include `teamId` ownership so multi-team beta does not require a rewrite.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent

1. **Start a session:** Give your coding agent the phase prompt at the beginning of each phase.
2. **Read selectively:** Each phase lists the exact PRD and vision sections needed for that phase.
3. **Let it work:** The agent reads the roadmap, finds the first unchecked task, implements it, and marks it complete.
4. **One session = one phase when possible:** If you stop mid-phase, resume from the first unchecked task.
5. **Push a PR for review:** When a phase is complete, push a PR and let a review agent such as CodeRabbit review it before starting the next phase.
6. **Need more context?** If a task references a section not in the phase references, read only that section on demand.

### Session Tips

- Do not skip tasks; they are ordered to avoid dependency gaps.
- Verify after each phase by running the app locally.
- Update the status line after completing tasks.
- Keep visual styling simple until `docs/design.md` exists.
- Treat scoring tests as product requirements, not optional cleanup.
